Staff Management Module — Technical Blueprint

1. Module Overview & Objectives
The Staff Management module governs every employee working at a property on the Pinesphere Stay platform
— from the property Owner down to Housekeeping and Security staff. It sits inside the Property Owner App
(Flutter, offline-first) and is one of the modules that must continue to function fully without internet, because
staff attendance, task assignment and role enforcement happen throughout the day on the ground, regardless
of connectivity.
Per the BRD, this module is directly responsible for: Attendance, Leave, Salary, Roles, Performance and
Task Assignment (Page 10 — Staff Management), and it feeds the Role-Based Permission engine referenced
in the Security section and the User Hierarchy diagram (Owner → Manager / Reception / Accountant →
Housekeeping / Maintenance / Restaurant → Guest).
The module has four responsibilities:
• Identity & Role Management — create staff profiles, assign one of the BRD-defined roles (Owner,
Manager, Reception, Cashier, Housekeeping, Maintenance, Restaurant, Security, Auditor, Accountant) and
bind granular module-level permissions to each role.
• Day-to-day Operations — attendance capture, leave application/approval, task assignment (e.g. a
housekeeping task tied to a room, or a maintenance complaint), and performance review.
• Payroll Support — monthly salary computation, payment status tracking and salary history, feeding the
Accountant role and the Expense/Accounting module.
• Governance — every create/update/delete/login action by staff is written to an immutable audit log, and
every staff device is bound and controllable from the Device Management module and Super Admin's Remote
Device Control.
Design Principle: Staff Management must obey the same offline-first contract as the rest of the app — every read
and write happens against the local SQLite/ObjectBox store first, and is queued for the Synchronization Engine to
push/pull once internet is detected.

2. Staff Roles Defined in the BRD
The BRD explicitly names the following operational roles under a property (Page 26 — Staff Management,
Page 18 — User Hierarchy, Pages 27–29 — Owner/Staff Login permissions):
Role

Scope (as per BRD)

Owner

Single primary owner per property. Full control: create staff, change prices, manage subscription,
manage devices, view all reports, approve refunds, manage settings.

Manager

Operational authority delegated by Owner; broad access across bookings, rooms, staff and reports (per
Role Matrix, Page 32-33).

Reception

Create booking, check-in, check-out, collect payments, view rooms. Cannot delete data, change

Cashier

Payment collection focused role (referenced alongside Reception/Accountant in staff role list).

Housekeeping

Cleaning, laundry, room status updates, lost & found.

Maintenance

Complaint tracking for furniture, electrical, plumbing, AC, TV, fan.

Restaurant

Order taking, kitchen status, billing, stock/inventory deduction (Addon module).

Security

Named as a staff role in the BRD; scoped for premise/guest safety oversight.

Auditor

Named as a staff role in the BRD; read-focused, aligned to Audit Log visibility.

Accountant

Payments, expenses, GST, reports, invoices.

Design decision: roles are modeled as data (a roles table), not hard-coded enums. The ten roles above are
seeded as system roles at property creation time, but the Owner can still create custom roles later — this
keeps the module flexible for the Enterprise/Custom subscription plans mentioned on Pages 15–16.

3. Core Logic Design
3.1 Role & Permission Enforcement Logic
Every module screen and API endpoint checks a permission tuple (role_id, module_name, action) before
allowing an action. This permission set is downloaded and cached locally on first login so checks work fully
offline. The Owner login is hard-restricted: only one staff row per property may have is_primary_owner =
true.

3.2 Attendance Logic
Staff punch in/out using Biometric, PIN or Manager-marked entry (Manual). A staff member can have only one
attendance record per calendar date (enforced by a unique constraint on staff_id + date). If no check-in exists
by end of business day and the day is not on approved leave, a background job (running locally, then
reconciled on sync) marks the day Absent. Half-day is derived when check-in and check-out span less than a
configurable minimum working duration.

3.3 Leave Management Logic
A leave request moves through Pending → Approved/Rejected, approvable only by a role holding
can_approve on the Staff module (typically Owner/Manager). Leave balance is computed on demand:
allocated days (from leave_types.max_days_per_year) minus the sum of approved leave days for the current
year. Approved leave dates automatically create corresponding attendance rows with status On-Leave so the
Attendance Dashboard never shows a false Absent.

3.4 Salary Logic
Net salary is computed as: Basic + Allowances + Overtime − Deductions − Advance Adjusted. One salary
record is allowed per staff, per month, per year (unique constraint). Salary generation is restricted to
Owner/Accountant. Payment status transitions Pending → Partially Paid → Paid, and every status change is
captured in the audit log because it is a financial action.

3.5 Task Assignment Logic
Tasks can originate from another module (e.g. a Housekeeping task auto-created at guest check-out, or a
Maintenance complaint logged against a room) or be created manually by a Manager. Each task stores an
optional related_module and related_record_id so it can deep-link back to the originating
room/booking/complaint. Overdue tasks (due_date passed, status still Pending/In-Progress) are automatically
flagged for the Owner/Manager dashboard.

3.6 Performance Logic
Performance is captured as periodic reviews (rating 0–5) tied to a review window, plus two derived,
system-calculated signals that feed the AI daily summary mentioned in the BRD (Page 15): attendance
consistency (% present) and task completion rate (% completed on time). These derived metrics are computed
at read-time from staff_attendance and staff_tasks — they are not stored redundantly.

3.7 Staff–Device Binding Logic
A staff member logs in from a registered device. A session row is created on login and closed on logout; the
currently active session determines whether the Super Admin's Remote Device Control (Page 25–26) can
force-logout that staff member instantly on next connectivity. This directly reuses the Device Management
module rather than duplicating device data.

3.8 Offline Sync & Conflict Resolution Logic
Every staff-related row carries version, updated_at and sync_status. On reconnect, the Sync Engine
(Page 4 of the BRD) pushes local changes and pulls cloud changes. Non-critical fields (e.g. remarks, photo)
use last-write-wins by updated_at. Financially or operationally sensitive fields — salary amounts, role
changes, permission changes, leave approval status — are flagged conflict instead of auto-merged, and
surfaced to the Owner for manual resolution, in line with the BRD's explicit 'Conflict Resolution' step in the sync
flow (Page 3–4).

3.9 Audit Logging Logic
Every create, update, delete, login, approval and rejection on a staff-related entity writes one immutable row to
staff_audit_log, storing old_value/new_value as JSON snapshots. The BRD is explicit: "Nothing can be deleted
from the audit log" (Page 31) — so this table has no delete permission at the application layer, and destructive
SQL operations are blocked at the database-role level too.

6. Database Design — Entities & Relationships
The Staff Management module introduces 12 entities. All tables exist both in the local offline store
(SQLite/ObjectBox, per device) and the cloud store (PostgreSQL), with the same logical schema — this
symmetry is what makes the Sync Engine's incremental push/pull straightforward.
Entity

Purpose

staff

Master record for every employee of a property (incl. the Owner).

roles

Defines a role (system-seeded or custom) scoped to a property.

role_permissions

Module-level CRUD/approve/export permission grants per role.

staff_attendance

One row per staff per calendar date.

leave_types

Lookup of leave categories (Casual, Sick, Paid, Unpaid, Emergency).

staff_leave

Leave applications and their approval lifecycle.

staff_salary

Monthly payroll record per staff.

staff_performance

Periodic performance review entries.

staff_tasks

Task assignment & tracking, optionally linked to another module's record.

staff_documents

ID proofs and HR documents with verification status.

staff_sessions

Login/logout session history, tied to a registered device.

staff_audit_log

Immutable log of every sensitive staff-related action.

Key Constraints
• Exactly one staff row per property may have is_primary_owner = true (partial unique index).
• employee_code and mobile_number are unique per property_id, not globally — the same phone
number can belong to staff at two different properties in a multi-tenant SaaS.
• staff_attendance enforces one row per staff per date.
• staff_salary enforces one row per staff per (month, year).
• staff_leave enforces from_date ≤ to_date via a CHECK constraint.
• staff_audit_log has no DELETE grant at the database-role level — matching the BRD's 'Nothing can be
deleted from the audit log' rule.
• All tables carry sync_status, version and timestamps to support the Incremental Sync + Conflict
Resolution features from the Synchronization Engine (BRD Page 4).

10. Security & Compliance Considerations
• Local database encryption for the SQLite/ObjectBox store, per BRD Page 15.
• Biometric login with PIN fallback for staff punch-in and app access.
• Role-based permissions enforced both client-side (UX gating) and server-side (API authorization) — never
trust the client alone.
• Device registration — a staff member can only log in from a device approved under that property (Device
Management module).
• Remote deactivation — Super Admin or Owner can revoke a staff member's session; local data is erased
from that device on next reconnect if the staff is terminated.
• Audit trail immutability — enforced at the database role level (REVOKE DELETE on staff_audit_log).
• Encrypted sync for all staff payloads travelling between device and cloud.
• PII minimization — Aadhaar/Passport numbers stored in staff_documents only, not duplicated into the staff
master row.

11. Scalability & Flexibility Notes
• Data-driven roles — roles and permissions live in tables, not code, so a new role (e.g. 'Night Auditor') can
be added per property without an app release.
• Multi-tenant safe uniqueness — all uniqueness constraints (mobile number, employee code) are scoped
by property_id, so the same schema serves Starter homestays and Enterprise multi-property chains alike.
• Polymorphic task linkage — staff_tasks.related_module / related_record_id lets the task engine attach to
any future module (e.g. Inventory restock task) without schema changes.
• Plan-aware limits — Maximum Staff Users per subscription plan (BRD Page 23) can be enforced by a
simple count check against the staff table at creation time, without new tables.
• Derived, not duplicated, metrics — performance scores are computed from attendance/task tables at read
time, keeping the schema lean and avoiding stale cached numbers.
• Sync-ready by design — every table already carries the version/sync_status/timestamps needed for
incremental sync, so this module plugs into the existing Synchronization Engine without special-casing.
• Backend framework — FastAPI's Pydantic models map 1:1 to these tables, keeping request/response
validation and the offline payload contract in sync as the schema evolves.
