1. Module Overview
1.1 Purpose & Scope
The User & Role Management module is the security and access-control backbone of the Pinesphere Stay platform. It
governs who can log in to the system, what each person is allowed to see and do, and how that access is created,
changed, revoked, and audited — across the Super Admin Portal, the Property Owner App, and every staff login
within a property. Because Pinesphere Stay is offline-first, this module must also work with zero internet connectivity:
a receptionist must be able to log in, and the app must be able to enforce role permissions, even when the property's
device has been offline for days.
This module does not duplicate business logic that belongs to other modules (Bookings, Payments, Housekeeping,
etc.). Instead, it is the single source of truth for identity (who is this user) and authorization (what is this user allowed
to do), which every other module queries before allowing an action.

1.2 Objectives
• Provide a secure, offline-capable login mechanism for Owners and Staff (PIN, biometric, password).
• Maintain a clear, hierarchical role structure exactly as defined in the BRD: Super Admin → Regional Manager /
Sales / Support / Property → Property Owner → Manager / Reception / Accountant → Housekeeping /
Maintenance / Restaurant → Guest.
• Enforce that every property has exactly one Primary Owner account, with all other logins created as Staff under
that Owner.
• Support a fixed catalogue of system roles (Owner, Manager, Reception, Cashier, Housekeeping, Maintenance,
Restaurant, Security, Auditor, Accountant) while remaining flexible enough to support custom, property-specific
roles in future.
• Guarantee that role and permission data is fully available offline and synchronises safely (with conflict handling)
whenever connectivity returns.
• Feed every user/role change into the platform-wide Audit Log module, since the BRD states nothing can be
deleted from the audit log.
• Integrate cleanly with the Device Management module (device-user binding, remote logout/lock) and the
Subscription module (plan limits on number of users/staff).

1.3 Position Within the Overall System
Per the BRD, Pinesphere is built as three applications sharing one multi-tenant FastAPI backend: the Super Admin
Portal (Pinesphere operations), the Property Owner App (Flutter, offline-first), and the Guest Portal (WhatsApp /
PWA, cloud-only). The User & Role Management module has a footprint in the first two:
• Super Admin Portal — manages Pinesphere-internal roles (Regional Manager, Sales Team, Support Team) and
has global oversight of every property's users, plus the ability to remotely lock, log out, or revoke any user's
access.
• Property Owner App — the Owner manages their own Staff (Manager, Reception, Housekeeping, Maintenance,
Restaurant, Accountant, Cashier, Security, Auditor) entirely offline; changes sync to the cloud when internet is
available.
• Guest Portal — Guests are not "users" of this module in the authentication sense (no password/PIN); they only
receive secure, time-boxed WhatsApp links. Guest identity/consent is out of scope for this module and is
handled by the Guest Portal / Booking modules.

4. Core Logics
This section defines the exact business logic the Flutter app and FastAPI backend must implement for this module.
Each logic block is written so it can be translated almost directly into function/service names during development.

4.1 Role-Based Access Control (RBAC) Enforcement Logic
Every screen, button, and API endpoint is guarded by a single reusable check rather than scattered if/else conditions:
• hasPermission(userId, permissionCode, requiredLevel) is the single choke-point function, implemented
identically on the Flutter client (for UI hide/show + offline enforcement) and the FastAPI backend (for hard
server-side enforcement).
• On login (online or offline), the full role_permissions snapshot for the user's role is loaded once into an inmemory PermissionSet and cached locally — this avoids a DB/query round trip on every tap.
• The client uses the cached PermissionSet to enable/disable UI (e.g. grey out "Delete Booking" for Reception);
the server is the final authority and re-checks on every API call — the client-side check is a UX convenience,
never the security boundary.
• Access levels are ordered (NONE < VIEW < OWN < LIMITED < FULL) so a single comparison (userLevel >=
requiredLevel) covers most checks; special cases (e.g. "OWN" meaning "only records created by me") add a rowlevel filter (created_by = current_user_id) on top of the level check.

4.2 Offline Authentication Logic
Because the app must run with zero connectivity, authentication cannot depend on a live call to the server at every
login.
• First login for any device must happen online (or via a Support-generated activation code) — this is when the
credential hash, role, and permission snapshot are downloaded and cached locally.
• Subsequent logins check credentials against the locally stored, encrypted hash: PIN (4–6 digit) hashed with a
salted algorithm (e.g. Argon2/bcrypt) or biometric assertion validated via the OS keystore/secure enclave —
never plain-text PIN storage.
• A password/PIN is never transmitted or re-validated against the server on every login; the server is only recontacted opportunistically (background sync) to detect remote revocations.
• If the cached credential set is missing or corrupted (e.g. app reinstalled with no local backup) and there is no
internet, login is denied with a clear "connect to the internet once to restore access" message rather than a
silent failure.

4.3 Single Primary Owner Enforcement Logic
• Each property may have exactly one user row with is_primary_owner = true. This is enforced at the database
layer with a partial unique index (unique on property_id where is_primary_owner = true), not only in application
code.
• Ownership can move (BRD: "Transfer Ownership", performed by Super Admin) via a dedicated
transfer_ownership(propertyId, fromUserId, toUserId) operation that, in a single transaction, flips
is_primary_owner off the old owner and on for the new one, and writes two audit_logs entries.
• The primary Owner account can never be deleted directly — only deactivated as part of a Transfer Ownership or
Property Offboarding operation initiated by Super Admin.

4.4 Multi-Tenant User Scoping Logic
• Every query for users, roles (custom ones), and sessions is automatically scoped by property_id at the repository
layer (both in the FastAPI service layer and the local SQLite/ObjectBox repository) — a developer should not be
able to accidentally query across properties.
• System roles (Owner, Manager, Reception, etc.) are global rows with property_id = NULL and are shared/readonly reference data; only custom, property-specific roles carry a property_id.
• Pinesphere Super Admin, Regional Manager, Sales and Support roles are the only ones allowed a NULL
property_id on the users table itself (i.e., a login not tied to any single property).

4.5 Custom Role Creation Logic (Scalability Hook)
• V1 ships only the fixed system roles from Section 2, but the roles/role_permissions schema already supports
property-scoped custom roles so this can be turned on later without a schema migration.
• Logic: an Owner (with USERS = FULL) may clone an existing system role into a new custom role row
(is_system_role = false, property_id = <own property>), then adjust individual permission rows in
role_permissions.
• Custom roles are always evaluated with the same hasPermission() function as system roles — no separate code
path — which keeps the feature additive rather than a fork.

4.6 Session & Token Management Logic
• Online session: FastAPI issues a short-lived JWT access token + longer-lived refresh token on successful
/auth/login; refresh happens silently in the background when the app is online.
• Offline session: a locally-signed offline session token is created after successful local PIN/biometric verification,
valid until the earlier of (a) a configurable max-offline-duration (e.g. 30 days, configurable per Subscription plan)
or (b) the next successful online sync.
• Every session is recorded in user_sessions with the originating device_id, so Device Management and Super
Admin can see exactly which device is logged in as which user.
• Logout (manual or remote, from Super Admin's "Remote Device Control") immediately invalidates the session
row; on the next sync/heartbeat the device honours the remote logout even if it is mid-offline-session.

4.7 Sync & Conflict Resolution Logic for User/Role Data
• All local writes to users, roles, role_permissions go through a generic outbox pattern: the write is applied locally
immediately, and a corresponding row is appended to user_sync_log with status = PENDING.
• When connectivity returns, the Synchronization Engine (shared platform service, per BRD) pushes pending rows,
then pulls server-side changes (new roles, permission changes made from Super Admin, remote deactivations).
• Conflict rule for this module specifically: Owner/Super Admin edits always win over Staff-originated edits (Staff
cannot edit their own role/permissions in the first place, so true conflicts are rare); for concurrent Owner edits
from two devices, Last-Write-Wins by updated_at timestamp, with the losing change preserved in audit_logs for
traceability rather than silently discarded.
• A locally created staff account that has not yet synced is flagged is_pending_sync = true in the UI so the Owner
knows that account will not yet appear on other devices or in Super Admin.

4.8 Account Lockout & Security Logic
• Configurable max failed PIN/biometric attempts (default 5) locks the account locally (status = LOCKED) and logs a
SECURITY_ALERT entry in audit_logs.

• A locked account can be unlocked by: (a) the Owner from the Staff Dashboard, (b) Support/Super Admin
remotely, or (c) automatically after a cool-down window, per configuration.
• Repeated lockouts across multiple staff on the same device within a short window raise a fraud/unusual-activity
flag surfaced on the Security & Audit Dashboard (Section 7.6) and to the AI Features' "Fraud and unusual activity
alerts".

4.9 Staff Invitation & Onboarding Logic
• Owner creates a staff_invitations row (mobile number + role) rather than a live user directly; an activation
code / WhatsApp link (reusing the platform's WhatsApp Integration) is generated for the staff member to set
their own PIN on first login.
• If created while offline, the invitation is queued exactly like any other write (Section 4.7) and the WhatsApp
message is actually sent only once the device/server syncs and has connectivity.
• Invitation expires after a configurable window; expired, unused invitations do not create a user row and are
periodically cleaned up.

4.10 Device–User Binding Logic (integration with Device Management)
• A user_devices row links a user to one or more registered devices; the BRD's Device Management screen
("Current User", "Last Login") reads directly from this table rather than duplicating device/user linkage
elsewhere.
• When Super Admin performs "Revoke License → Logout → Erase Local Database → Disable Login" for a resigned
staff member's device, this module's response is: invalidate all user_sessions rows for that device, set the user's
status to INACTIVE if it was their only device, and write the mandatory audit_logs entry.

6.5 Offline (SQLite/ObjectBox) vs Cloud (PostgreSQL) Strategy
• All tables in Section 6.3 exist in both stores except user_sync_log's cloud mirror (used for audit/traceability only)
and any Pinesphere-internal-only rows (Super Admin/Sales/Support users are not replicated to property devices
at all).
• Local IDs are generated client-side as UUIDs (not auto-increment integers) so records created offline never
collide with cloud-generated IDs from another device.
• Sensitive columns (pin_hash, password_hash) are stored encrypted at rest locally using the device's secure
keystore-backed encryption key, per the BRD's "Local database encryption" security requirement.
• ObjectBox is used where relational joins are simple and read-speed for permission checks matters most (users,
roles, role_permissions, permissions); SQLite is acceptable for lower-frequency tables (staff_invitations,
credential_reset_requests, user_sync_log) — final choice is an implementation detail behind a repository
interface, so it can change without affecting the rest of the app.

6.6 Indexing & Performance
• INDEX on users(property_id, status) — used by every Staff Dashboard listing query.
• INDEX on users(mobile_number) — used for login lookup and duplicate-guest/staff detection.
• INDEX on role_permissions(role_id) — used on every permission-set load at login.
• INDEX on user_sessions(user_id, expires_at) — used to quickly find/expire active sessions.
• INDEX on user_sync_log(sync_status, created_at) — used by the background Sync Engine to fetch the pending
outbox in order.
• INDEX on audit_logs(property_id, created_at DESC) — used by the Audit Logs module dashboards; kept here
since this module is a heavy writer to that table.

9. Scalability & Flexibility Considerations
• Custom roles per property: the schema (Section 6.3.1) already supports property-scoped roles alongside global
system roles, so enabling the Role & Permission Configuration Dashboard's "create fully custom role" capability
later requires no migration.
• Granular, extensible permission engine: new permission codes (e.g. a future module) are just new rows in
permissions plus corresponding role_permissions grants — no schema change, and hasPermission()
automatically supports them.
• Multi-property staff (future): the current UNIQUE(property_id, mobile_number) design allows the same person
to hold independent staff accounts at multiple properties today; a true "single identity, multiple property
memberships" model can be introduced later via a join table (user_property_memberships) without breaking
existing data, since users already carries a nullable property_id.
• Pluggable authentication: password_hash/pin_hash columns are deliberately generic; adding SSO/OAuth for
Super Admin or corporate-chain Owners later is an additive column (e.g. auth_provider, external_id), not a
redesign.
• Horizontal scale of the backend: FastAPI is stateless per request (session state lives in user_sessions/JWTs, not
server memory), so multiple API instances can sit behind a load balancer; PostgreSQL is the single source of
truth and can be scaled with read replicas for the heavy-read dashboards in Section 7.
• Sync engine reusability: the outbox pattern (user_sync_log) used here is the same pattern the BRD implies for
every other offline module (Bookings, Payments, Housekeeping, etc.) — implementing it once as a shared core
service (lib/core/sync/) means this module's sync logic is a thin, reusable case rather than bespoke code.
• Dashboard growth: each dashboard in Section 7 is backed by its own read endpoint, so adding a 7th dashboard
later (e.g. a Franchise/Regional Manager rollup) is additive and does not touch the other five.

10. Security Considerations (Module-Specific)
Directly mapped from the BRD's platform-wide Security section, scoped to what this module is responsible for
implementing:
• Local database encryption — pin_hash, password_hash, and any cached PermissionSet data are encrypted at
rest on-device.
• Biometric login with PIN fallback — both paths funnel into the same offline authentication logic (Section 4.2);
neither is a separate, weaker code path.
• Device registration and activation — every login is tied to a registered device via user_devices; unregistered
devices cannot bootstrap offline credentials (Section 6.3.5).
• Role-based permissions — enforced identically client-side (UX) and server-side (real boundary), per Section 4.1.
• Remote device deactivation — Super Admin/Support can revoke a device's sessions the moment it reconnects,
per Section 4.10.
• Audit logs for all critical actions — every create/edit/deactivate/role-change/reset/lockout event in this module
writes an audit_logs row; the table is append-only at the database-permission level (Section 6.3.10).
• Secure backups — user/role data included in the platform's secure backup routine (owned by the Support
Portal, this module supplies the data).
